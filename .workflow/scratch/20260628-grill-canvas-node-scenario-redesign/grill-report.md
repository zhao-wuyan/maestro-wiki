# Grill Report: Canvas Node Scenario Redesign

**Session**: GRL-20260628-canvas-node-scenario-redesign
**Depth**: standard (5 branches)
**Date**: 2026-06-28T00:00:00Z
**Upstream**: user proposal in conversation

## Discovery Summary

### Project Context

The project is an independent Vite + React Maestro Workflow Wiki. M1/Phase 1 is implemented and validated, but the user is dissatisfied with the current layout and interaction model. The new proposal prefers a large canvas with an initial scene-selection node, then a branching node graph where each selected command fixes a node and creates the next node to the right.

### Codebase Surface

- `src/main.tsx:66` defines `ScenarioModel` as one complete pre-authored flow with `nodes`, `edges`, and `steps`.
- `src/main.tsx:78` defines only `unclearRequirementsScenario`, so there is not yet a scene-category selector.
- `src/main.tsx:309` stores `selectedNodeId` and `activeStepId`, supporting one active flow rather than a growing user-built graph.
- `src/main.tsx:353` renders a fixed SVG with predefined node coordinates.
- `src/main.tsx:404` renders a separate `<aside className="detail-panel">`, which conflicts with the user's request to avoid a large right sidebar.
- `src/styles.css:78` defines `.workspace` as `grid-template-columns: minmax(0, 1fr) 390px`, creating the large right panel.

### Upstream Material

User wants the product reframed around a large canvas. The center initial node should ask users to select a scenario type, such as new project initialization, new requirement implementation, source exploration/Q&A, or adding tests. Those informal examples should be refined into Maestro-defined task categories. After scenario selection, the UI recommends initial commands. After command selection, the first node is fixed and connected to a next node on the right, which presents primary and secondary command options.

### Q1.1: Canvas rewrite boundary

**Question**: Current code uses `src/styles.css:78` as a two-column `1fr + 390px` layout and `src/main.tsx:404` as a large `<aside className="detail-panel">`. The user wants a large canvas instead. How far should the rewrite go?
**Answer**: User wants a canvas rewrite, but asked what that concretely means and how deep the rewrite goes.
**Evidence**: `src/styles.css:78` hardcodes the right panel column; `src/main.tsx:404` owns the panel. User explicitly rejected the large sidebar.
**Decision**: open
**Constraint**: The canvas rewrite boundary MUST be clarified before planning implementation.

### Q1.2: Primary and secondary command definition

**Question**: If a next node shows primary and secondary commands, are those manually authored labels or generated by a rule engine?
**Answer**: User challenged the distinction: primary/secondary command selection sounds rule-based, so the deeper rule-engine option may be necessary.
**Evidence**: Current `src/main.tsx:60` has `choices` and `src/main.tsx:61` has `alternatives`, but the current model does not encode ranking rationale or rules. It only stores authored options.
**Decision**: open
**Constraint**: The data model MUST distinguish authored command options from rule-ranked command recommendations.

### Q1.3: Rule source from maestro-flow

**Question**: How should the local recommendation rules be generated from `maestro-flow` content rather than invented manually?
**Answer**: User confirmed the canvas + local rule recommendation direction is correct, but identified source-derived rule generation as the hard part.
**Evidence**: Source inventory exists in `maestro-flow/.claude/commands/maestro-*.md` and `maestro-flow/guide/*.md`. Current app only stores manually authored `choices` and `alternatives` in `src/main.tsx:60-61`, with no extracted rule provenance.
**Decision**: open
**Constraint**: Recommendation rules SHOULD be derived from command definitions and guide documents with explicit provenance, not from opaque hand-authored UI text alone.

### Q1.4: maestro-next as recommendation reference

**Question**: Can the canvas recommendation model reuse `maestro-next` instead of inventing a new command-ranking scheme?
**Answer**: User pointed to `maestro-next` and `maestro-help` as likely references for handling many command recommendations.
**Evidence**: `maestro-flow/.claude/commands/maestro-next.md:100` maps project state to lifecycle position and natural next command; `maestro-flow/.claude/commands/maestro-next.md:128` defines scoring signals; `maestro-flow/.claude/commands/maestro-next.md:139` defines candidate preconditions; `maestro-flow/.claude/commands/maestro-next.md:157` defines an intent-to-command routing table; `maestro-flow/.claude/commands/maestro-next.md:188` defines workflow clusters.
**Decision**: locked
**Constraint**: Canvas recommendation rules MUST start from `maestro-flow/.claude/commands/maestro-next.md` lifecycle, scoring, precondition, routing-table, and workflow-cluster concepts before adding UI-specific refinements.

### Q1.5: maestro-help as catalog and workflow reference

**Question**: Can `maestro-help` supply the command catalog and workflow path taxonomy for the canvas start node?
**Answer**: User clarified that `maestro-help` exists in the Codex skills, not only as a CLI command-help helper.
**Evidence**: `maestro-flow/.codex/skills/maestro-help/SKILL.md:39` declares `catalog.json` as the single data source; `maestro-flow/.codex/skills/maestro-help/catalog.json:7` lists commands with category, description, and source; `maestro-flow/.codex/skills/maestro-help/catalog.json:158` defines workflow topology and paths; `maestro-flow/.claude/skills/maestro-help/phases/03-workflow-guide.md:17` defines task type patterns and workflow paths.
**Decision**: locked
**Constraint**: Canvas start-node scenario taxonomy SHOULD reuse `maestro-help` catalog/workflow data: command categories, essential commands, task patterns, workflow topology, and Path A-F definitions.

### Q1.6: Scope synthesis

**Question**: Does the redesigned MVP primarily replace the right-panel scenario page with a rule-backed canvas, rather than simply restyling the old flow?
**Answer**: Yes. The accepted direction is canvas + local rule recommendation engine, with `maestro-help` and `maestro-next` as upstream rule references.
**Evidence**: User confirmed the direction after discussing `maestro-next` and `maestro-help`; current code conflict remains `src/styles.css:78` and `src/main.tsx:404`.
**Decision**: locked
**Constraint**: The next implementation MUST be treated as a product-model redesign, not a cosmetic layout-only refactor.
## Branch 2: Data Model & State

**Status**: In Progress

### Q2.1: Upstream-to-UI data boundary

**Question**: Current `src/main.tsx:66` uses one authored `ScenarioModel`; `maestro-help` has catalog/path data and `maestro-next` has scoring/preconditions. How should upstream content become UI data?
**Answer**: Generate a stable rule table first.
**Evidence**: `maestro-flow/.codex/skills/maestro-help/catalog.json:7` provides commands/categories/source; `maestro-flow/.codex/skills/maestro-help/catalog.json:158` provides workflow topology/paths; `maestro-flow/.claude/commands/maestro-next.md:126` defines scoring concepts; current `src/main.tsx:66` lacks a reusable rule layer.
**Decision**: locked
**Constraint**: The canvas UI MUST consume a project-local generated or curated rule table, not parse upstream markdown directly at render time.

### Q2.2: Rule update ownership

**Question**: If the local rule table is generated from `maestro-flow`, who or what performs the review/update step after upstream changes?
**Answer**: The update should be LLM-assisted, not manual. The user wants a repeatable protocol so an AI assistant can accurately update rules each time.
**Evidence**: `maestro-flow/.codex/skills/maestro-help/catalog.json:7` is structured enough for extraction; `maestro-flow/.claude/commands/maestro-next.md:126` contains semi-structured scoring/routing rules but still needs interpretation. Current app has no update protocol or generated rule artifact.
**Decision**: locked
**Constraint**: Rule updates MUST be governed by an explicit LLM update protocol: fixed source priority, extraction schema, diff checks, ambiguity handling, validation tests, and provenance requirements.

### Q2.3: Canvas path persistence

**Question**: Canvas nodes are generated from user choices. Current `src/main.tsx:309` stores only `selectedNodeId` and `activeStepId`, so state resets on refresh. Where should generated nodes and route history live?
**Answer**: Persist route history in browser storage. By default keep the latest 10 route maps; users can favorite routes, and favorites have no count limit, are not overwritten, and are not automatically cleared. The phrase "报错前10条路线图" is treated as likely meaning "保存前10条路线图" and needs confirmation.
**Evidence**: Current state is volatile React state at `src/main.tsx:309`; no `localStorage` or persistence layer exists. User explicitly requested browser persistence and favorites.
**Decision**: locked with wording confirmation needed
**Constraint**: The canvas state model MUST include browser-persisted route history with capped recent routes and uncapped protected favorites.

### Q2.4: Favorite route payload

**Question**: What should a favorite route map save: only command chain, recommendation evidence, or full canvas snapshot?
**Answer**: Save recommendation evidence.
**Evidence**: A command-only chain would lose why a command was primary; a full UI snapshot would bind data to layout details. Current `ScenarioStep` already separates command, condition, purpose, choices, alternatives, and citations at `src/main.tsx:51`, so the model can evolve toward saving recommendation evidence without storing exact SVG positions.
**Decision**: locked
**Constraint**: Saved routes MUST store scenario choice, selected command nodes, primary/secondary recommendations, recommendation reasons, sourceRefs, name, and createdAt; they SHOULD NOT require persisting exact canvas coordinates for MVP.

### Q2.5: Rule update protocol artifact

**Question**: Where should the LLM-governed rule update protocol live so future updates are repeatable?
**Answer**: It should become a project protocol/spec that the AI assistant must follow when updating rules from `maestro-flow`.
**Evidence**: User requested a repeatable rule so updates through the assistant stay accurate. Current project has `.workflow/specs/` and docs, but no dedicated recommendation-rule protocol yet.
**Decision**: locked
**Constraint**: The next plan SHOULD add a rule update protocol document before or alongside the rule table implementation.

### Q3.1: Ambiguous recommendation handling

**Question**: If rules cannot identify one clear primary command, how should the canvas show uncertainty?
**Answer**: Show multiple same-level recommendations and do not force a primary command.
**Evidence**: `maestro-flow/.claude/commands/maestro-next.md:246` defines close-call behavior for top1/top2 score gaps; current app only has one active `command` per `ScenarioStep` at `src/main.tsx:54`, so the next model must support ties or groups.
**Decision**: locked
**Constraint**: The recommendation model MUST support equal-rank candidate groups and an uncertainty state; the UI MUST NOT fabricate a primary command when rule confidence is insufficient.

### Q3.2: Recent history typo confirmation

**Question**: The user wrote "默认报错前10条路线图" while describing saved routes. How should this be interpreted?
**Answer**: Treat as "默认保存前 10 条路线图" unless corrected.
**Evidence**: Surrounding context discussed browser persistence, favorites, overwrite rules, and cleanup, not errors.
**Decision**: locked with low risk
**Constraint**: Recent non-favorite route history SHOULD keep the latest 10 route maps; favorites are excluded from this cap.

### Q5.1: Canvas expansion strategy

**Question**: With 50+ commands and many workflow paths in `maestro-help`, should the canvas expand step by step, show a full path preview, or compare multiple paths?
**Answer**: Use stepwise expansion, and allow running multiple branches.
**Evidence**: `maestro-flow/.codex/skills/maestro-help/catalog.json:7` contains a large command catalog; `maestro-flow/.codex/skills/maestro-help/catalog.json:168` lists multiple workflow paths. **Decision**: locked with branch behavior open
**Constraint**: Canvas MUST expand recommendations incrementally from the current node; unselected branches SHOULD NOT all become full nodes immediately. Multi-branch behavior needs a dedicated interaction rule.

**Decision**: locked
