# 用户指令记忆规则

## 描述

当用户在对话中教导或指示 AI 助手如何执行任务或以某种方式行动时，这些教导应被记录到仓库根目录的 `.monkeycode/MEMORY.md` 文件中。此外，当 Agent 在执行任务过程中发现有价值的项目知识，也应主动将这些发现记录到同一文件中。

### 记录内容的核心原则：记录"怎么做"，而非"做了什么"

MEMORY.md 只记录用户对模型**行为方式**的指令（即"应该怎么做某件事"），不记录用户让模型完成的**具体任务**（即"做了什么"）。

正确示例（记录行为方式）：
- "每次改完代码后跑 `npm run lint`"——这是行为指令
- "这个项目用 pnpm，不要用 npm"——这是行为指令

错误示例（记录具体任务）：
- "用户让我修复了登录页面的 bug"——这是任务记录，不应写入
- "完成了用户管理模块的 CRUD 接口"——这是任务记录，不应写入

---

## 排除项（严禁记录）

**以下内容绝对不允许写入 MEMORY.md。在每次记录前，必须逐条核对以下排除项，命中任意一条即放弃记录。**

### 1. 设计和代码实现上的要求

开发过程中用户对 UI 布局、页面结构、组件模式、代码架构、实现方式等的要求。这些要求已经落实到代码中，通过阅读代码即可获取，不需要额外记忆。

> 判断依据：这条信息能否通过阅读代码获知？例如——页面用了几列网格布局、某个组件应该从哪个共享模块读取元数据、提交前要做字段映射、后端用了什么架构模式——这些都在代码里，不需要记。

### 2. 某次开发任务的具体实现细节

功能实现记录、技术细节、代码架构说明、终验结果等属于"做了什么"的范畴。代码本身就是最好的文档。

> 判断依据：这条内容是在描述"本次开发做了什么"还是"以后每次应该怎么做"？前者不记，后者可记。

### 3. 已存在于其他文档中的内容

如果内容已经写入 `.monkeycode/docs/`、`协作文档/`、`工程开发进度.md` 或其他项目文档中，不要再重复记录到 MEMORY.md。MEMORY.md 不是文档索引。

> 判断依据：这条信息是否已经在项目的其他文档中有完整记录？如果有，MEMORY.md 不需要再存一份。

### 4. 产品命名和品牌表达决策

产品命名、品牌文案、视觉表达风格等属于产品文档范畴，不属于 MEMORY.md 的记录范围。

> 判断依据：这条内容是不是关于"产品叫什么、产品怎么说、产品长什么样"？如果是，属于产品文档。

### 5. 可通过阅读代码直接获取的信息

接口定义、类型声明、函数签名、模块路径、文件路径、常量定义、代码逻辑流程等。这些信息通过 Explore 代码即可获取。

> 判断依据：如果另一位 Agent 用 Grep/Glob 工具在代码库中搜索就能找到答案，就不要记录。

### 6. 显而易见的内容

项目使用了什么语言、什么框架等不需要特别记忆的基础信息。

---

## 允许记录的内容范围

MEMORY.md **只记录**以下类别的内容：

| 类别 | 说明 | 正确示例 |
|------|------|---------|
| **运维部署** | 服务器地址、部署目录、服务名称、健康检查地址、部署流程 | "线上服务器公网 IP 为 xxx，SSH 用户为 ubuntu" |
| **构建编译** | 构建命令、构建工具、构建验证步骤 | "前端构建：`cd /workspace/business-toolkit && npm run build`" |
| **排错调试** | 排障工作流、调试方法论、常见问题排查步骤 | "排查工具不可用时按'前端页面->路由映射->后端定义->认证->接口复测'链路逐层确认" |
| **工作流协作** | 多 Agent 协作规范、开发流程、提交策略、进度文档制度 | "不同 Agent 必须分工负责不同模块，每次修改后立即 git add + commit + push" |
| **环境配置** | 开发环境的特殊配置、环境变量、工具可用性 | "本环境 apply_patch 工具不存在，编辑文件请使用 Write/Edit 工具" |
| **行为指令** | 用户对助手行为方式的直接指令（回复语言、操作约束等） | "所有思考过程和回复必须使用中文" |

**判断标准**：在记录前问自己——"后续另一位 Agent 进入项目时，这个信息能否帮助他快速上手运维、构建、排错和协作？"如果答案是"能"，记录；如果答案是"读代码就能知道"或"代码里已经写了"，不记录。

---

## 实现方式

- 监控对话中用户的教导性指令
- 在任务执行过程中，主动识别并记录项目特有的**运维、构建、排错、工作流、环境配置**知识
- **记录前必须逐条核对上述排除项**，命中任意一条则放弃记录
- 添加新条目前，先扫描 `.monkeycode/MEMORY.md` 检查是否存在相似或相同的指令
- 若发现重复，跳过新条目或与已有条目合并
- 合并时，更新已有条目的上下文或日期
- 若无重复，将指令追加到仓库根目录的 `.monkeycode/MEMORY.md` 中
- 条目格式应包含日期、用户指令和上下文
- 在记录有用的行为指令时注意保护用户隐私
- **文件大小管理**：更新 MEMORY.md 时，如果文件内容已经超过 150 行，需要将相同模块/类别的规则合并到一条规则里，避免 MEMORY.md 过大。合并时保留关键信息，去除冗余描述

## 目的

建立用户偏好和指令的持久化记忆，用于指导未来的交互和定制。

## 强制执行事项

- 如果 [记忆文件](.monkeycode/MEMORY.md) 不存在，**请在首次尝试记录内容时**，带上本次需要记录的内容，按照**文件模板**创建该文件。
- 如果 [记忆文件](.monkeycode/MEMORY.md) 存在，**必须**在首次回复前读取 [记忆文件](.monkeycode/MEMORY.md) 作为项目级指令！
- **每一条**用户输入都应按照以下**触发条件**进行验证，一旦匹配，**必须立即**先核对排除项，确认不在排除范围内后，再更新 [记忆文件](.monkeycode/MEMORY.md)

## 触发条件

### 1. 用户提供关于行为方式的明确指令

> 示例：
> - "回复我的时候请用中文"
> - "Always respond in bullet points"
> - "代码注释统一用英文写"
> - "不要在回复中使用 emoji"

### 2. 用户指示或纠正助手的行为

> 示例：
> - **满足某条件时做某事**："每次改完代码后自动跑一下 lint"、"When I ask you to refactor, always write unit tests first"
> - **满足某条件时不做某事**："不要自动删除注释掉的代码"、"Don't modify files outside the src/ directory unless I explicitly say so"
> - **在某项目/模块中使用（或不使用）某工具**："这个项目用 pnpm 不要用 npm"、"Use `pytest` instead of `unittest` in this repo"、"在 backend 模块里不要用 print，用 logger"

### 3. 用户提出偏好的实现方式

> 示例：
> - "我更喜欢函数式写法，少用 class"
> - "Prefer composition over inheritance in this project"
> - "写 SQL 的时候用 CTE 而不是子查询"
> - "CSS 优先用 Tailwind 的 utility class，不要写自定义样式"

**代码风格记录的约束**：尽量不要记录一般的代码风格偏好（如缩进、命名风格等），除非用户明确且强烈要求（例如 "以后都用函数式风格写 React 组件，必须遵循这个原则"）。代码风格应在项目配置文件中定义（如 `.eslintrc`、`.prettierrc`），MEMORY.md 聚焦于行为指令和无法从代码中获取的项目知识。

### 4. 用户说明期望的任务执行方式

> 示例：
> - "改 bug 之前先帮我写个能复现问题的测试用例"
> - "每次提交代码前先跑 `make check`"
> - "重构的时候一次只改一个文件，改完我确认了再继续"
> - "When adding a new API endpoint, always update the OpenAPI spec first"

### 5. Agent 在执行任务过程中主动发现项目知识

Agent 在执行任务过程中，如果发现了对后续**运维、构建、排错、协作**有参考价值的项目知识，应主动记录到 MEMORY.md 中。这类知识不需要用户明确指示，Agent 应自行判断并记录。

#### 触发时机

- **执行构建时**：了解到项目的构建命令、构建工具链、构建配置等
- **运行测试时**：了解到测试框架、测试命令、测试约定等
- **代码生成时**：发现项目使用了代码生成工具（如 protobuf、OpenAPI codegen、ORM migration 等），记录生成的时机和方法
- **调试问题时**：发现项目的特殊依赖关系、环境要求、已知陷阱等
- **部署运维时**：了解到服务器信息、部署流程、健康检查方式等

#### 记录格式

使用与用户指令相同的条目格式，但 Context 字段标注为 Agent 自主发现：

```
[项目知识摘要]
- Date: [YYYY-MM-DD]
- Context: Agent 在执行 [具体任务描述] 时发现
- Category: [运维部署|构建方法|测试方法|排错调试|工作流协作|环境配置]
- Instructions:
  - [具体的知识点，逐行描述]
```

#### 记录前检查清单

在记录 Agent 发现的项目知识前，必须确认：

1. 是否属于运维部署/构建编译/排错调试/工作流协作/环境配置范畴？（不属于则不记录）
2. 是否已经写入了其他项目文档？（已写入则不重复记录）
3. 是否可通过阅读代码直接获取？（可获取则不记录）
4. 是否是某次开发任务的具体实现细节？（是则不记录）
5. 是否已有相似条目存在于 MEMORY.md？（有则合并或跳过）
